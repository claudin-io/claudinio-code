❯ Gostaria de que o handoff do brain para o builder fosse uma nova sessão mesmo 
  e o primeiro prompt fosse o plano numa especie de handoff e aqui tem uma      
  explicacao de como seria um bom handoff:                                      
  https://www.youtube.com/watch?v=dtAJ2dOd3ko&t=415s Com base no vídeo de Matt  
  Pocock, a técnica de "handoff" (transferência) é uma habilidade criada para   
  lidar com as limitações de sessões longas com agentes de IA (como Claude Code 
  ou Copilot). O objetivo principal é comprimir o contexto atual da conversa    
  em um arquivo Markdown para que uma nova sessão de IA possa continuar o       
  trabalho de forma focada e inteligente [00:26].                               
  Aqui está uma explicação detalhada de como fazer e estruturar um handoff:     
  Por que fazer um Handoff?                                                     
  À medida que você conversa com um agente de IA e ele faz chamadas de          
  ferramentas ou edita arquivos, a janela de contexto fica cada vez mais cheia. 
  Eventualmente, você entra no que o Matt chama de "zona burra" (dumb zone),    
  onde o agente começa a dar respostas piores porque a sua atenção está muito   
  difusa devido ao excesso de tokens [02:07]. O handoff permite que você inicie 
  uma nova sessão (na "zona inteligente") sem perder o progresso.               
  Como fazer um Handoff (Regras e Estrutura)                                    
  Quando você pedir ao seu agente de IA para criar o documento de handoff, ele  
  deve seguir estas diretrizes:                                                 
  Defina o propósito da nova sessão: Toda vez que você pedir um handoff, diga   
  explicitamente ao agente por que você está fazendo isso e qual será o foco da 
  próxima sessão. O agente usará isso para adaptar o documento especificamente  
  para a próxima tarefa [11:13].                                                
  Crie uma seção de "Habilidades Sugeridas" (Suggested Skills): O documento     
  deve listar quais ferramentas ou habilidades o novo agente precisará invocar  
  logo de cara (ex: ferramentas de diagnóstico ou leitura de documentação)      
  [09:54].                                                                      
  Não duplique conteúdo: O handoff não deve conter informações que já existem   
  em outros arquivos, código ou issues do GitHub. Em vez de copiar tudo, o      
  agente deve usar apenas ponteiros e referências para esses arquivos [10:28].  
  Salve em uma pasta temporária: Os arquivos de handoff são descartáveis. Peça  
  ao agente para salvá-los no diretório temporário do seu sistema operacional,  
  e não no seu espaço de trabalho atual, para evitar que apodreçam como         
  documentação inútil no seu repositório [10:48].                               
  Remova informações sensíveis: Certifique-se de que o agente omita chaves de   
  API, senhas ou dados pessoais (PII) do documento.                             
  Padrões de Uso Comuns                                                         
  O vídeo destaca duas maneiras muito úteis de usar o handoff no seu fluxo de   
  trabalho:                                                                     
  Separar tarefas paralelas (Manter a sessão pura): Se você está focado em uma  
  tarefa e percebe um bug ou oportunidade de refatoração fora do escopo, não    
  faça isso na mesma sessão (o que diluiria o contexto). Em vez disso, peça ao  
  agente para criar um handoff apenas sobre esse bug e passe esse documento     
  para um agente em uma janela separada. Assim, ambos podem trabalhar de forma  
  independente [05:45].                                                         
  O Padrão de Prototipagem (Sub-agente DIY): Ao planejar uma feature complexa e 
  você se deparar com algo que precisa ser testado na prática (como uma nova    
  UI), você faz um handoff para uma nova sessão focada em construir um          
  protótipo. Depois que a sessão do protótipo for concluída e você aprender o   
  que precisava, você faz um handoff de volta com os aprendizados para a sua    
  sessão de planejamento original [08:30].                                      
  Por ser apenas um documento Markdown simples, você também pode usar o handoff 
  para transferir o contexto entre diferentes ferramentas e IA's (ex: passar    
  do Claude Code para o GitHub Copilot) de forma muito simples.

     Linked Sessions: handoff Brain→Builder e handoff por limite de tokens

     Context

     Hoje Brain (planejamento) e Builder (execução) são dois modos da mesma 
     sessão — a transição é um flip no ModeCtl, e o system prompt é reconstruído
     a cada round. Isso significa que o Builder herda todo o contexto (já
     poluído) da fase de planejamento, e sessões longas entram na "dumb zone"
     descrita por Matt Pocock, mitigada hoje apenas por auto-compactação aos
     150k tokens.

     Mudança desejada (decisões confirmadas com o usuário):

     1. Brain→Builder = nova sessão: toda transição (botão "Continue with
     Builder", tool exit_plan_mode, golden loop) cria uma sessão nova cujo
     primeiro prompt é composto pelo harness apontando para o plano
     (.claudinio/plans/*.md) + task list. Não precisa de handoff gerado por LLM
     — o plano é suficiente. O golden loop é simétrico: Builder→Brain também
     cria sessão nova.
     2. Handoff por limite de tokens: quando o contexto passa de um limite
     configurável (slider nas Settings, 120k–256k, default 120k), o modelo
     comprime o próprio contexto num documento de handoff (estilo Matt Pocock:
     propósito, estado, decisões, ponteiros — nunca conteúdo duplicado —,
     trabalho em andamento, próximos passos, sem segredos/PII) e uma nova sessão
     do mesmo modo continua de onde parou, com o handoff como primeiro prompt.
     O handoff substitui a compactação; a compactação vira fallback (falha de
     geração, perfil GitSync). Se o slider > 150k, o limiar de compactação sobe
     junto.
     3. UX contínua: para o usuário é uma conversa só. Registros
     linked_from/handoff_to no jsonl encadeiam as sessões; a UI renderiza a
     cadeia como um thread único e a lista de sessões mostra uma entrada por
     cadeia.

     1. Mecanismo central: RunOutcome + transição única

     session.rs

     pub enum HandoffReason { PlanExecution, GoldenFlip, ContextThreshold,
     ManualBuilder }

     pub struct HandoffSpec {
         pub reason: HandoffReason,
         pub next_mode: SessionMode,
         pub next_origin: ModeOrigin,
         pub first_message: String,          // primeiro prompt da nova sessão
         pub golden_cycle: u32,              // estado do golden loop a carregar

     pub enum RunOutcome { Completed, Handoff(Box<HandoffSpec>) }
     run_workflow/run_workflow_with_profile passam a retornar Result<RunOutcome,
     String>. Caminhos de handoff escrevem Done + Status + HandoffTo na sessão
     antiga mas não emitem AgentEvent::Done (a conversa continua para a UI).

     Novo módulo src-tauri/src/agent/transition.rs

     link_session(ws, modes, steering_map, records_cache, old_handle, spec, 
     event_tx) — função única usada por todos os gatilhos:
     1. Append SessionRecord::HandoffTo { next_session_id, reason, ts } na store
     antiga.
     2. SessionStore::create nova; append SessionRecord::LinkedFrom { 
     prev_session_id, reason, golden_cycle, golden_stalls, golden_last_pending, 
     ts } logo após Meta.
     3. Copia BaseCommit (via earliest_base_commit) para a nova store (sem
     re-ancorar HEAD).
     4. Copia snapshot de tasks (load_last_tasks → append_tasks), golden tasks
     inclusas.
     5. Append Mode { next_mode, next_origin }; insere novo ModeCtl em
     state.modes sob o novo id; remove o antigo.
     6. Move o Arc<SteeringCtl> do id antigo para o novo (steering enfileirado
     sobrevive).
     7. Troca ws.active_session.
     8. Emite AgentEvent::SessionLinked { prev_session_id, session_id, reason, 
     mode, first_message }.

     Também neste módulo: compose_builder_kickoff() (plano + tasks, ASCII para
     passar reject_non_english), compose_golden_kickoff(),
     compose_context_handoff_message().

     Driver loop em commands/agent.rs

     Extrair o corpo do tokio::spawn de send_message para um helper
     spawn_run_loop que faz loop:
     - Completed → break.
     - Handoff(spec) → link_session, history = vec![], message = 
     spec.first_message, rebuild ToolContext (novo store_path; base_commit da
     cadeia), mesmo event_channel → continua o loop (streaming contínuo).
     - Err → tratamento de erro existente.

     state.mode_for vira função livre (ou fn sobre os maps clonáveis) para poder
     ser chamada dentro da task spawnada. Perfil GitSync sempre retorna
     Completed (checks de handoff guardados por profile == Standard).

     Config

     Checagem (só em fronteira de round — nunca há tool call pendente)

     Nos dois pontos onde a compactação já checa (:1139 e :1299):
     1. est = estimate_tokens(...); se profile == Standard && est >= threshold →
     geração de handoff.
     2. Senão, se est >= compact_threshold_efetivo → compactação existente
     (fallback). compact_threshold_efetivo = max(COMPACT_THRESHOLD, threshold + 
     10k) para o slider > 150k não ser atropelado pela compactação.
     3. Guarda de MAX_CONTEXT_TOKENS inalterada.

     Geração do handoff (turno forçado só-texto, sem tool novo)

     1. Emite marker TextStep("__handoff_start__:...") (espelha
     __compact_start__).
     2. push_user_blocks com instrução [system]: responder apenas o documento de
     handoff — seções ## Purpose of next session, ## Current state, ## Key 
     decisions, ## Pointers (paths e task ids, nunca conteúdo duplicado), ## 
     In-flight work, ## Next actions; sem segredos/PII; sem tools.
     3. Uma chamada stream_message_with_retry com tools = &[] (texto é a única
     saída possível).
     4. Sucesso: persiste o Turn, append SessionRecord::Handoff { text, ts }
     (auditoria), Done+Status, marker __handoff_done__, retorna Handoff { 
     ContextThreshold, mesmo modo, golden state, first_message = wrapper do 
     harness com o texto + "call tasks_get first" + path do plano }.
     5. Fallback (erro após retries ou texto vazio 2x): marker __handoff_fail__
     e cai na compact_history existente — nunca trava a sessão. Sanity check
     mínimo (tamanho/headings) antes de aceitar o doc.
     6. Interrupt do usuário durante a geração → abandona o handoff, caminho de
     interrupção normal.
