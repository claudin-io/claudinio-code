# Git Changes Panel — Solution Design

## Context / Problem Statement

Adicionar um indicador de git no header do ChatPanel (mesma linha de "New" e "History") que mostra o estado do working directory, com modal para visualizar changes e atalho para commit & push via agente.

## Solution Design

### 1. Backend: novo módulo `commands/git.rs`

Criar dois novos comandos Tauri no arquivo `src-tauri/src/commands/git.rs`:

#### a) `git_status(workspace: String) -> GitStatus`
Executa `git -C <root> status --porcelain` no workspace ativo e parseia a saída. Retorna:

```
GitStatus {
  has_changes: bool,
  files: Vec<ChangedFile>,
  total_additions: u32,
  total_deletions: u32,
}
```

Onde `ChangedFile` tem:
- `path: String` — caminho relativo ao workspace
- `status: String` — "M" (modified), "A" (added), "D" (deleted), "??" (untracked), "R" (renamed)
- `additions: u32` — linhas adicionadas (via `git diff --numstat`)
- `deletions: u32` — linhas removidas (via `git diff --numstat`)

Para untracked files, `additions/deletions` são estimados via `wc -l` (diff não funciona sem staged add).

Files que são "both modified" (conflito de merge) tratados como modified.

#### b) `git_file_diff(workspace: String, path: String) -> String`
Executa `git -C <root> diff <path>` e retorna o diff unificado completo.

#### c) `git_branch(workspace: String) -> String`
Executa `git -C <root> rev-parse --abbrev-ref HEAD` para mostrar o nome da branch atual (usado no tooltip do indicador).

#### Registro em `lib.rs`:
Adicionar `commands::git::git_status`, `commands::git::git_file_diff`, `commands::git::git_branch` ao `generate_handler!` e expor o módulo.

### 2. Frontend: IPC bindings (`src/lib/ipc.ts`)

Adicionar três novas funções:

```typescript
export interface ChangedFile {
  path: string;
  status: string;  // "M" | "A" | "D" | "??"
  additions: number;
  deletions: number;
}

export interface GitStatus {
  hasChanges: boolean;
  files: ChangedFile[];
  totalAdditions: number;
  totalDeletions: number;
}

export function gitStatus(workspace: string): Promise<GitStatus>
export function gitFileDiff(workspace: string, path: string): Promise<string>
export function gitBranch(workspace: string): Promise<string>
```

### 4. Componente GitIndicator (src/components/GitIndicator.tsx)

**Props:**
- `workspace: string` — caminho do workspace ativo
- `onSendCommitPush: () => void` — callback para acionar commit & push

**Comportamento:**

1. **Polling a cada 5s** via `setInterval` → `gitStatus(workspace)`.
2. **Polling do branch** a cada 30s via `gitBranch(workspace)`.
3. **Estado visual:**
   - `hasChanges === false`: texto `"0 changes"` em `text-ink-faint` com opacidade reduzida (opacity-50), ícone git-branch. Botão não clicável? Na verdade, sempre clicável para mostrar modal mesmo vazia.
   - `hasChanges === true`: `"N changes (+A −D)"` em `text-ink-muted` com cor normal. Clicável.
4. **Tooltip:** mostra `"branch-name · N files changed"`.
5. **Layout:** `<div class="flex items-center gap-1 px-2 py-1 text-[11px]">` — mesmo padding e font-size dos botões vizinhos.

### 6. Fluxo Commit & Push

Quando o usuário clica em "Commit & Push":
1. Modal de changes fecha
2. `newSession(workspace)` é chamado para criar nova sessão
3. `sendMessage(workspace, "Please commit and push all changes", [])` é chamado para iniciar o agente
4. O chat panel mostra o agente trabalhando — sem input visível (é um novo session com mensagem já enviada, então o input fica vazio)
5. O agente usa `git add`, `git commit`, `git push` (com aprovação do usuário quando necessário)

### 7. Integração no ChatPanel

No header do ChatPanel (`src/components/ChatPanel.tsx`), depois do botão "History":

```tsx
<div class="flex items-center gap-1">
  <button onClick={startNewSession}>...</button>
  <ContextWarning ... />
  <button onClick={toggleSessions}>...</button>
  <GitIndicator workspace={props.workspace} onSendCommitPush={handleCommitPush} />
</div>

<GitChangesModal
  workspace={props.workspace}
  open={showGitModal()}
  onClose={() => setShowGitModal(false)}
  onSendCommitPush={handleCommitPush}
/>
```

`handleCommitPush`:
```typescript
async function handleCommitPush() {
  setShowGitModal(false);
  // Switch to a new session automatically
  await newSession(props.workspace);
  // The input will be empty, so the user just sees "agent is thinking"
  // Actually we want to auto-send the message. But sendMessage needs the session event loop setup.
}
```

Hmm, preciso pensar nisso com cuidado. O `sendMessage` do ChatPanel é chamado através do `handleSend`. Preciso expor uma forma de enviar mensagem automaticamente.

**Alternativa:** Em vez de auto-enviar, faço o seguinte:
1. Crio uma nova sessão
2. Coloco a mensagem "Please commit and push all changes" no estado `input` do chat
3. E aciono o `handleSend` programaticamente

Ou mais simples: faço o `startNewSession()` e depois uso a função `sendMessage` do IPC diretamente (que está em `ipc.ts`), passando o canal de eventos.

Vou criar o fluxo assim: o `GitChangesModal` recebe um callback `onCommitPush()`. No ChatPanel, esse callback:
1. Chama `newSession(props.workspace)`
2. Chama `sendMessage(props.workspace, "Please commit and push all changes", [])` 
3. Isso inicia a sessão e envia a mensagem — o painel de chat já está configurado para escutar eventos

## Risks & Edge Cases

1. **Workspace sem git:** `git_status` retorna `{ hasChanges: false, files: [] }`. Indicador mostra "0 changes" normalmente — só não tem branch name.
2. **Muitas changes (milhares de arquivos):** `git diff --numstat` pode ser lerdo. O status --porcelain é rápido. Para o diff individual, só carregamos quando o usuário expande.
3. **Grandes diffs:** O diff preview limita a ~200 linhas iniciais? A ideia é mostrar o diff completo mas com scroll no container. Se o arquivo tiver diff enorme, o scroll interno da modal resolve.
4. **Conflitos de merge:** Aparecem como modified, mas `git diff` vai mostrar os marcadores de conflito.
5. **Polling:** 5s para changes, 30s para branch. Cleanup do interval no `onCleanup` do componente.
6. **Mensagem duplicada:** Se o usuário clicar "Commit & Push" várias vezes, cria sessions duplicadas. Desabilitar o botão após o primeiro clique.
