# v0.1.3 Release Plan

## Context / Problem Statement
Precisamos fazer o deploy da versão `v0.1.3` do Claudinio Code. A release atual é `v0.1.2` e há 11 commits no `main` desde então. O workflow de CI/CD (`.github/workflows/release.yml`) dispara ao fazer push de uma tag `v*` e builda para Windows (x64), macOS (arm64) e Linux (x64), criando uma GitHub Release automaticamente.

## Goal (Definition of Done)
- Versão `v0.1.3` publicada como GitHub Release em https://github.com/claudin-io/claudinio-code/releases
- Workflow de CI executado com sucesso (verde) no Actions: https://github.com/claudin-io/claudinio-code/actions
- Release notes geradas automaticamente a partir dos commits entre `v0.1.2` e `v0.1.3`

## Key Findings (Prova Real)
1. **Workflow dispara por tag**: `.github/workflows/release.yml:4-6` — triggers: push de tags `v[0-9]+.[0-9]+.[0-9]+` e `[0-9]+.[0-9]+.[0-9]+`
2. **Versão em 3 arquivos**: `package.json:4` (`"version": "0.1.2"`), `tauri.conf.json:5` (`"version": "0.1.2"`), `Cargo.toml:3` (`version = "0.1.2"`)
3. **Release usa softprops/action-gh-release@v2**: suporta `generate_release_notes: true` (nativo do GitHub) e `body` customizado
4. **11 commits desde v0.1.2**: `git log v0.1.2..HEAD --oneline` mostra 11 commits no `main`
5. **Remote**: `origin git@github.com:claudin-io/claudinio-code.git`

## Authoritative Inputs
- Versão target: `0.1.3` (confirmado pelo usuário)
- Release notes: dinâmicos via `git log` entre a tag anterior e atual (confirmado pelo usuário)
- Estratégia de geração: step adicional no workflow (dinâmico, confirmado pelo usuário)

## Changes (Steps)

### Step 1: Bump version to 0.1.3
- **Target**: `package.json:4`, `src-tauri/tauri.conf.json:5`, `src-tauri/Cargo.toml:3`
- **Mutation**: `"0.1.2"` → `"0.1.3"` nos 3 arquivos
- **Why**: Versão nova da release
- **Constraints**: Alterar APENAS o número da versão, nada mais

### Step 2: Update release.yml para release notes dinâmicos
- **Target**: `.github/workflows/release.yml` — step `Create Release`
- **Mutation**: 
  - Adicionar step anterior ao `Create Release` que gera release notes via `git log` entre a tag anterior (v0.1.2, v0.1.3, etc.) e a atual
  - O script deve: detectar a tag anterior com `git tag --sort=-v:refname | head -2 | tail -1`, gerar `git log <prev_tag>..HEAD --oneline --no-merges`, formatar como Markdown
  - Passar o resultado para o step `Create Release` como `body`
- **Why**: Release notes dinâmicos que evoluem automaticamente a cada release

### Step 3: Commit, tag, and push
- **Target**: repositório `main`
- **Mutation**: `git add` dos 3 arquivos de versão + workflow, `git commit`, `git tag v0.1.3`, `git push origin main --tags`
- **Why**: Disparar o workflow de release via push da tag
- **Constraints**: Commit atômico, sem `git add -A`

### Step 4: Verify CI execution
- **Target**: GitHub Actions
- **Mutation**: Nenhuma — apenas verificar que o workflow iniciou e completou com sucesso
- **Why**: Critério de done: release finalizada e CI verde

### Lições aprendidas
1. **Shallow checkout pegou!** `actions/checkout@v4` faz shallow fetch (depth=1) por padrão. O step de release notes rodou sem as tags anteriores, gerando "Initial Release" com só 1 commit. Fix: adicionar `fetch-depth: 0` e `fetch-tags: true` no checkout do job `create-release`.
2. As release notes da v0.1.3 foram corrigidas manualmente com `gh release edit`. Para releases futuras, o fix do workflow já garante que as notas serão geradas corretamente.
