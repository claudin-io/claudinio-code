# File Editor no Monaco (double-click no Browse Files)

## Context / Problem Statement

Atualmente, no painel "Browse Files" (`FileTree`), um **double-click** num ficheiro abre-o no programa padrão do SO (`onOpenExternal` → `openPath` via Tauri). O utilizador quer que o double-click abra o ficheiro diretamente no **Monaco Editor** dentro da app, podendo **editar, guardar (Ctrl+S)** e fazendo **auto-complete** (como VSCode) para ficheiros `.ts/.tsx/.js/.jsx`.

Já existe uma `TextEditorModal` com Monaco, mas é usada apenas no ChatPanel para editar o texto da mensagem. Vamos criar um novo componente `FileEditorModal` focado em edição de ficheiros do disco.

## Decision Log

| Decisão | Escolha |
|---|---|
| Double-click → Monaco vs external app | **Substituir**: double-click abre no Monaco editor |
| Edição read-only vs edição completa | **Edição completa**: pode editar e guardar (Ctrl+S / botão Save) |
| Aviso ao fechar com alterações não guardadas | **Sim**: diálogo de confirmação se dirty |
| Auto-complete | **Sim**: para .ts/.tsx/.js/.jsx com tipos do projeto |

## Solution Design

### 1. Novo componente: `FileEditorModal`

Criar `src/components/FileEditorModal.tsx` — baseado no `TextEditorModal` mas com:

**Props:**
```ts
interface FileEditorModalProps {
  filePath: string;
  rootPath: string;
  onClose: () => void;
}
```

**Behaviour:**
- **Ao montar**: lê o ficheiro via `readFile(filePath)` do `ipc.ts`
- **Linguagem**: detectada automaticamente pela extensão (Monaco `language` mapping)
- **Título**: mostra o nome do ficheiro + caminho relativo ao workspace
- **Botão Save** + **Ctrl+S**: guarda via `writeFile(filePath, content)`
- **Dirty state**: sinal que fica `true` quando o conteúdo do editor difere do original
- **Close com dirty**: se dirty, mostra um confirm dialog nativo `window.confirm()` antes de fechar
- **Auto-complete TS/JS**: configura `monaco.languages.typescript.typescriptDefaults` com:
  - Compiler options: ESNext target, Node module resolution, JSX preserve, strict
  - Extra libs: carrega ficheiros `.ts/.tsx/.js/.jsx` do projeto e regista como `addExtraLib()` para auto-complete cross-file

**Layout:**
- Modal quase-fullscreen (10% margin)
- Header: file path + botão Save + botão Close
- Monaco editor no restante espaço

### 2. Modificar `FileTree.tsx`

- Alterar `handleDblClick` para não chamar `onOpenExternal` mas sim chamar `onOpenFile` (ou uma nova callback `onOpenInEditor`)
- Ou melhor: criar uma nova prop `onOpenInEditor: (path: string) => void` e chamá-la no double-click, mantendo `onOpenExternal` para uso futuro

Na verdade, como a decisão foi "substituir: double-click abre sempre no Monaco", podemos simplesmente mudar o `handleDblClick` para chamar `props.onOpenFile` em vez de `props.onOpenExternal`. Mas isso perderia a capacidade de abrir externamente. Melhor: manter `onOpenExternal` mas deixar de ser chamado no double-click. Em vez disso, o `App.tsx` deixará de passar `openExternal` como `onOpenExternal`.

**Alteração concreta:**
- `handleDblClick` em `TreeNode`: mudar de `props.onOpenExternal(props.entry.path)` para `props.onOpenFile(props.entry.path)`
- Opcionalmente, adicionar um menu de contexto ou botão para abrir externamente mais tarde

### 3. Modificar `App.tsx`

- Importar `FileEditorModal`
- Adicionar sinal `editorFilePath: () => string | null`
- No double-click do `FileTree`, em vez de `onOpenExternal`, chamar `setEditorFilePath(path)`
- Quando `editorFilePath()` não é null, mostrar `<FileEditorModal>` com o path
- **Single-click**: mantém o comportamento atual de `onOpenFile={setSelectedFile}` (highlight)

### 4. I18n

Adicionar keys aos locale files:

```ts
// en-US
"fileEditor.title": "{fileName} — Editor",
"fileEditor.save": "Save",
"fileEditor.unsavedTitle": "Unsaved changes",
"fileEditor.unsavedMessage": "You have unsaved changes. Are you sure you want to close?",

// pt-BR
"fileEditor.title": "{fileName} — Editor",
"fileEditor.save": "Salvar",
"fileEditor.unsavedTitle": "Alterações não guardadas",
"fileEditor.unsavedMessage": "Tem alterações não guardadas. Tem a certeza que deseja fechar?",
```

### 5. Auto-complete (TS/JS cross-file types)

Para o auto-complete funcionar como VSCode, o Monaco precisa de conhecer os tipos dos outros ficheiros do projeto. Estratégia:

1. Configurar `monaco.languages.typescript.typescriptDefaults.setCompilerOptions()` com opções que espelham o `tsconfig.json` do projeto
2. No `onMount` do `FileEditorModal`, carregar a lista de ficheiros `.ts/.tsx/.js/.jsx` do workspace e registá-los como `extraLibs` via `monaco.languages.typescript.typescriptDefaults.addExtraLib()`

Isto permite que o auto-complete resolva imports relativos (ex: `import { something } from "./otherFile"`) e forneça type hints.

**Nota de performance:** carregar todos os ficheiros como extraLibs de uma vez pode ser pesado para projetos grandes. A alternativa é carregar apenas quando o editor abre, e apenas os ficheiros do workspace atual. Para projetos com >1000 ficheiros podemos considerar lazy loading, mas para já a versão simples é suficiente.

## Risks / Considerations

- **Performance com muitos ficheiros TS**: carregar todos os ficheiros `.ts/.tsx` como extraLibs pode ser lento em projetos grandes. Monitorizar e otimizar se necessário (ex: carregar só ficheiros referenciados pelo ficheiro aberto).
- **Ficheiros grandes**: o Monaco lida bem, mas ficheiros >1MB podem causar lag.
- **Guardar em disco**: usar `writeFile` do Tauri — se falhar, mostrar erro ao utilizador.
- **O Monaco já está no bundle**: não há dependências novas para adicionar.

**Journal:** **Key decisions and learnings from implementing the Monaco File Editor:**

1. **Separate component, not reuse of TextEditorModal**: The existing TextEditorModal takes an initial text string, while the file editor reads from disk and needs file system I/O (readFile/writeFile). The props and lifecycle are fundamentally different, so a new FileEditorModal was the right call.

2. **Language detection from extension**: Simple ext-to-language map covering 10+ extensions. Monaco falls back to "plaintext" for unknown types, which works well.

3. **Monaco already in bundle**: No extra dependency needed — `monaco-editor@^0.55.1` was already a dep. No webpack plugin required; Monaco loads its workers automatically from the CDN when configured as a direct import.

4. **TS auto-complete via extraLibs**: To provide intellisense for project files, we load up to 200 `.ts/.tsx/.js/.jsx` source files from `fileIndexMap` and register them as `addExtraLib()`. This gives cross-file type awareness. Capped at 200 as a SIMPLIFICATION — large projects could be optimized by loading only the dependency graph or files under the current directory.

5. **Dirty state tracking**: `originalContent` is set once on initial file read; the dirty signal compares `editor.getValue()` against it on every model change. This avoids having to track individual edit operations.

6. **Unsaved changes dialog**: Uses `window.confirm()` (browser native) rather than a custom modal — intentionally minimal. YAGNI until a more sophisticated dialog is needed.

7. **Double-click replaces external app open**: The `onOpenExternal` prop on FileTree is preserved (sidebar still has context menu potential), but double-click now routes to the in-app editor via the new `onDblClickFile` prop.

8. **Edge cases handled**: Escape key closes (with dirty guard), overlay backdrop click closes, Ctrl+S / Cmd+S saves, dirty indicator shows `*` next to the filename in the header, and relative path shown on desktop widths.
