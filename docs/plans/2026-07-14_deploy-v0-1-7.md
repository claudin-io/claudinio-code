## Solution Design

The deployment follows the established release pipeline documented in the deploy skill (`.agents/skills/deploy/SKILL.md`):

1. **Bump version** from `0.1.6` тЖТ `0.1.7` in all 3 version files:
   - `package.json` (line 3: `"version": "0.1.7"`)
   - `src-tauri/Cargo.toml` (line 3: `version = "0.1.7"`)
   - `src-tauri/tauri.conf.json` (line 4: `"version": "0.1.7"`)

2. **Sync Cargo.lock** тАФ run `cargo check` in `src-tauri/` so Cargo regenerates the lockfile with the new version string. Without this, CI's `pnpm install --frozen-lockfile` will fail.

3. **Validate** тАФ `pnpm test` (~625 tests) and `pnpm build` must pass before tagging.

4. **Commit** тАФ `git add` the 4 changed files, commit with message `chore: bump version to 0.1.7`.

5. **Tag** тАФ create annotated tag `v0.1.7` with message `v0.1.7`.

6. **Push** тАФ push `main` and the tag to GitHub. The tag push triggers `.github/workflows/release.yml`, which builds the Tauri app for Windows (x64), macOS (ARM), and Linux (x64) and publishes a GitHub Release with installers.

**No changes to CI workflow, no feature changes, no config changes beyond version strings.**

## Risks

| Risk | Mitigation |
|---|---|
| CI build failure on a platform | CI runs 3 independent builds; a failure on one does not block others. Fix and re-tag if needed. |
| `pnpm install --frozen-lockfile` fails in CI | Prevented by running `cargo check` locally to sync Cargo.lock before commit. |
| Signing key missing from GitHub Secrets | Out of scope for this task тАФ CI uses `TAURI_SIGNING_PRIVATE_KEY` and `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`. |
| Tag already exists | Not applicable тАФ `v0.1.7` does not exist yet. |

## Non-goals

- No modification to the CI workflow (`.github/workflows/release.yml`)
- No feature changes or bug fixes (all 18 commits are already in `main`)
- No changelog file creation (CI generates release notes from git log automatically)
- No store submission (App Store, Microsoft Store) тАФ GitHub Release only

### Data flow

1. Version bump тЖТ `cargo check` regenerates `Cargo.lock` with new version
2. `pnpm build` (which runs `vitest run && vite build`) validates frontend
3. Git commit + tag `v0.1.7`
4. `git push origin main` + `git push origin v0.1.7`
5. Tag push triggers `.github/workflows/release.yml` (matches pattern `v[0-9]+.[0-9]+.[0-9]+`)
6. CI: `pnpm install --frozen-lockfile` тЖТ `pnpm tauri build` on 3 runners
7. CI: `softprops/action-gh-release@v2` creates GitHub Release with collected installers

### Step sequence (commands)

```bash
# Step 1: Bump version (edit 3 files)
# Step 2: Sync lockfile
cd src-tauri && cargo check && cd ..

# Step 3: Validate
pnpm test
pnpm build

# Step 4: Stage and commit
git add package.json src-tauri/Cargo.toml src-tauri/Cargo.lock src-tauri/tauri.conf.json
git commit -m "chore: bump version to 0.1.7"

# Step 5: Tag
git tag -a v0.1.7 -m "v0.1.7"

# Step 6: Push
git push origin main
git push origin v0.1.7
```

### Existing patterns reused

- Version bump follows the exact pattern from v0.1.6: same 4 files staged, same commit message format `chore: bump version to X.Y.Z`
- Tag format `vX.Y.Z` matches CI trigger regex and existing tag history
- Lockfile sync via `cargo check` (NOT `cargo build` тАФ faster, sufficient for lockfile update)

### Integration points

- **Vite build** (`vite.config.ts`): reads `APP_VERSION` from `package.json` at build time via `define` тАФ auto-updates when `package.json` changes
- **CI workflow** (`.github/workflows/release.yml`): triggered by tag push, no changes needed
- **GitHub Release** (`softprops/action-gh-release@v2`): uses tag name and version from CI, auto-generates release notes from git log between tags
