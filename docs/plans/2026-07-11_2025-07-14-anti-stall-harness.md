# Anti-Travamento do Harness (Context Overflow & Brain Stall)

## Context

A sessão `242f5357` travou num estado impossível de sair: Brain Mode ativado pelo usuário, o agente ignorou a entrevista obrigatória (`ask_user`), disparou 4 subagentes em paralelo + buscas adicionais, explodiu o contexto, e nunca produziu os entregáveis obrigatórios (plano + tasks). Como o `origin` era `"human"`, o agente não podia chamar `exit_plan_mode`.

O código atual tem **3 vulnerabilidades** que permitem esse padrão:

1. **Sem limite no size dos tool_results** — subagent reports e outputs de ferramentas vão para o histórico sem truncation. 4 subagentos podem devolver ~60k+ tokens de uma vez.
2. **Compact check só no início do loop** — `COMPACT_THRESHOLD` é verificado uma vez antes do `for` loop. Depois que subagentes e ferramentas enchem o contexto, não há re-check antes da próxima LLM call.
3. **Sem guard contra Brain spinning** — o agente pode gastar rounds infinitos em `spawn_agents` + `semantic_search` + `grep` sem nunca chamar `ask_user`, `write_plan` ou `tasks_set`.

## Solution Design

### A. Tool Result Truncation (máx. 6000 tokens)

Em `session.rs` `run_tool()`, depois de obter o `ContentBlock::tool_result`, truncar o conteúdo para um máximo de ~6000 tokens (≈24000 chars). Isso impede que um único tool_result exploda o contexto.

**Onde:** `session.rs` — logo antes do `return` no match `tools::execute()`.

### B. Pre-LLM Context Check (por round)

No início de cada iteração do `for _ in 0..max_rounds` em `run_workflow()`, re-estimar o contexto. Se estiver >= `COMPACT_THRESHOLD`, forçar compact_history antes de chamar `stream_message_with_retry`.

**Onde:** `session.rs` — dentro do loop principal, antes do bloco que chama `stream_message_with_retry`.

### C. Brain Mode Progress Guard

Trackear rounds consecutivos em Brain mode que só executam `spawn_agents`, `semantic_search`, `code_search`, `grep`, `read_file`, `file_outline`, `list_dir`, `symbol_lookup`, `go_to_definition`, `find_references`. Após N=4 rounds sem `ask_user`, `write_plan`, ou `tasks_set`, injetar um `[system]` reminder via `push_user_blocks`.

**Onde:** `session.rs` — no loop principal, após processar tool_results, antes de continuar para o próximo round.

## Risks

- **Tool Result truncation pode perder informação** — mitigado: truncamento é só no history block; os events do UI (AgentEvent::ToolResult com `content: String`) já são truncados a 2000 chars separadamente.
- **Brain guard pode ser falso positivo** — mitigado: 4 rounds de exploração é um limite generoso; qualquer `ask_user`, `write_plan` ou `tasks_set` reseta o contador.
- **Compact forçado no meio do loop pode perder estado** — mitigado: `compact_history` já existe e preserva o tail de 2+ turns do usuário; a mesma função já roda no início.

## Implementation Tasks

1. **ToolResult size cap** — `session.rs` truncate tool_result content > 24000 chars
2. **Pre-LLM context check per round** — `session.rs` re-check COMPACT_THRESHOLD before stream_message_with_retry
3. **Brain progress guard** — `session.rs` track explore-only rounds and inject reminder after 4

- Inseri o bloco entre o mode refresh e `stream_message_with_retry`, dentro do `for` loop
- O compact check existente (antes do loop) continua intacto — este é um segundo ponto de segurança
- Re-faz o rebuild do history + stats exatamente como o compact check do início
- Precisei também re-checkar o mode/system/tools após o compact, porque o compact pode mudar o mode (via golden loop)

### C — Brain progress guard
- Defini `BRAIN_EXPLORE_LIMIT = 4` rounds consecutivos
