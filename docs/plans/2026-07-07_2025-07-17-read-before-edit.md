# read-before-edit: Require read_file before edit_file

## Context

The `edit_file` tool currently accepts edits to any file without checking whether the agent has previously read that file. This means the agent can edit files it has never inspected, leading to blind or erroneous modifications. We need to enforce that `edit_file` only succeeds when the agent has called `read_file` on the same file first, optionally restricted to a line range.

## Requirements (confirmed via interview)

1. **Session-level in-memory tracking**: each `read_file` call records the path (and optional line range). Resets on new session.
2. **Only `read_file` counts**: not `file_outline`, `grep`, or other tools.
3. **Rejection with error**: when `edit_file` is called without prior `read_file`, it errors with `"read_file must be called on <path> before editing it"`.
4. **Range parameters on `read_file`**: `start_line` (1-based, optional) and `end_line` (1-based inclusive, optional). When absent → whole file.
5. **No range = full file access**: reading without range means any `old_string` in that file is valid for editing.
6. **Range overlap counts**: if `old_string` is found within any recorded range (even partially crossing a boundary), the edit is accepted.
7. **Multiple reads accumulate**: ranges from multiple `read_file` calls on the same file are unioned.
8. **TDD approach**: write tests first, then implement.

/// Tracks which files have been read and which line ranges were covered.
/// Thread-safe via the Mutex that already wraps ToolContext fields.
#[derive(Clone, Default)]
pub struct ReadTracker {
    /// Per-file record. `full_read: true` means the whole file was read
    /// (so any edit is allowed). `ranges` is a list of (start, end) 1-based
    /// inclusive line ranges from partial reads.
    pub files: HashMap<String, ReadFileRecord>,
}

#[derive(Clone, Default)]
pub struct ReadFileRecord {
    pub full_read: bool,
    pub ranges: Vec<(usize, usize)>,
}

### 7. Helper function to check if old_string is in range

Add a function in `edit_file.rs` or `mod.rs`:

```rust
/// Check if the old_string exists at a line position covered by the recorded
/// reads. Returns Ok(()) if the file was fully read or the old_string's line
/// falls within any recorded range, Err otherwise.
pub fn validate_read_before_edit(
    path: &str,
    old_string: &str,
    tracker: &ReadTracker,
) -> Result<(), String> {
    let record = tracker.files.get(path).ok_or_else(|| {
        format!(
            "read_file must be called on {} before editing it.\n\
             Call read_file first to read the file (or the relevant line range).",
            path
        )
    })?;
    
    if record.full_read {
        return Ok(());
    }
    
    // Find which line old_string starts on
    let content = std::fs::read_to_string(path)
        .map_err(|e| format!("cannot read {}: {e}", path))?;
    
    let line = content.lines()
        .position(|l| l.contains(&old_string.lines().next().unwrap_or("")))
        .map(|l| l + 1)  // 1-based
        .ok_or_else(|| "old_string not found in file".to_string())?;
    
    // Check if any range covers this line
    if record.ranges.iter().any(|(start, end)| line >= *start && line <= *end) {
        Ok(())
    } else {
        Err(format!(
            "read_file was called on {}, but only lines {}-{} were read. \
             The old_string starts at line {} which is outside the read range. \
             Call read_file again with the correct range.",
            path,
            record.ranges.iter().map(|(s, e)| format!("{s}-{e}")).collect::<Vec<_>>().join(", "),
            line,
        ))
    }
}
```

## Risks

1. **Thread safety**: `ToolContext` fields are not `Arc<Mutex<>>` like `read_tracker` needs to be. Check how `embedding_model` is handled (it's `Arc<Mutex<Option<SharedEmbedder>>>`). Using the same pattern: `Arc<Mutex<ReadTracker>>`.

2. **Existing tests**: All existing `ToolContext` construction sites in tests will need the new field. These are in `mod.rs` tests and `session.rs` tests.

3. **Agent workaround**: The agent could read the file, then the file changes externally before edit. The existing `apply()` function already re-reads the file and re-checks `old_string`, so this isn't a new risk — it's already handled.
