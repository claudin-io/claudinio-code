# Plan: Deploy Claudinio Chat v0.1.4

## Context

O usuário solicitou o deploy da versão v0.1.4 do Claudinio Chat. O projeto está atualmente na v0.1.3 com o HEAD em `2d8d716` (branch `main`, working tree clean). O deploy é feito via GitHub Actions: criar a tag `v0.1.4`, fazer push, e a release pipeline do `.github/workflows/release.yml` gera os builds para Windows/macOS/Linux e publica a GitHub Release.

Durante a investigação, identificamos 1 teste Rust falhando: `agent::tools::tests::test_read_file_large_truncated`. Ele espera um aviso de truncamento (`FILE SIZE WARNING`) que não é emitido porque o arquivo de teste (5000 linhas, ~10000 tokens) está abaixo do limite `MAX_TOKENS = 25000`.

O usuário optou por corrigir o teste para refletir o comportamento atual do código (aumentar o tamanho do arquivo para forçar truncamento) e seguir com o deploy.

## Solution Design

### Passo 1: Corrigir o teste Rust falhando

**Arquivo:** `src-tauri/src/agent/tools/mod.rs` — função `test_read_file_large_truncated`

**Problema:** O teste cria 5000 linhas. Cada linha "lineN" tem ~2-6 tokens em cl100k_base. Total = ~10000-15000 tokens, que é MENOR que `MAX_TOKENS = 25000`. Portanto, `truncate_by_tokens` retorna o conteúdo completo sem aviso.

**Correção:** Aumentar o número de linhas de 5000 para 30000. Com 30000 linhas × ~3-6 tokens/linha = ~90000-180000 tokens, o arquivo excede confortavelmente `MAX_TOKENS = 25000`, forçando o truncamento e a emissão do aviso.

- O conteúdo truncado terá ~6250-8333 linhas (~9 linhas de warning + conteúdo)
- O warning `FILE SIZE WARNING` aparecerá
- `start_line/end_line` será sugerido
- A contagem de linhas final será < 29000 (truncado) e > 10

## Risks

- O teste corrigido pode ser sensível à contagem exata de tokens (que varia entre implementações do tokenizer). Aumentar para 30000 linhas dá margem segura acima de `MAX_TOKENS`.
- CI pode falhar por razões ambientais (timeout de dependências, falta de secrets `TAURI_SIGNING_PRIVATE_KEY`). O workflow tem `fail-fast: false`, então cada plataforma é independente.
- Se as secrets de signing não estiverem configuradas no repositório, o build no macOS pode falhar (ou gerar binário não assinado).

This effectively deleted the `Determine version` step from the `create-release` job.

Fix was straightforward: add the missing blank line and `- name: Determine version` between the checkout's `with:` and the step metadata.
